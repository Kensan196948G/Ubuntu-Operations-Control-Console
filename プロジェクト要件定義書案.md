# 1. 要件定義書案

```md
# Ubuntu Ops Control Console 要件定義書

## 1. システム名

Ubuntu Ops Control Console

## 2. 目的

個人利用のUbuntuサーバ上で稼働する systemd サービス、Dockerコンテナ、Docker Composeプロジェクトを、WebUIから安全かつ見やすく制御・監視できる運用管理システムを構築する。

本システムは会社用途ではなく、個人の自宅サーバ・検証サーバ運用を対象とする。

## 3. 背景

Ubuntu上で複数のsystemdサービス、Dockerコンテナ、Docker Composeプロジェクトを運用している場合、状態確認や再起動、ログ確認のたびにSSH接続とCLI操作が必要になる。

本システムでは、日常運用で頻繁に使う以下の操作をWebUIに集約する。

- 稼働状態の確認
- サービス再起動
- コンテナ再起動
- Docker Composeプロジェクトの状態確認
- ログ確認
- 操作履歴確認

## 4. 利用者

個人利用者1名を想定する。

- Ubuntuサーバ管理者
- Docker利用者
- 自宅サーバ運用者

## 5. 認証方針

MVPではログイン認証は実装しない。

ただし、以下の制約を必須とする。

- インターネットへ直接公開しない
- 初期状態では localhost または LAN 内からのみアクセス可能とする
- 危険操作は禁止する
- 操作対象は allowlist に登録されたものだけとする
- すべての操作を監査ログへ記録する

## 6. 対象範囲

### 6.1 対象とするもの

- Ubuntuホストの基本状態表示
- systemd service の状態確認
- systemd service の start / stop / restart
- Dockerコンテナ一覧表示
- Dockerコンテナの start / stop / restart
- Dockerコンテナログ表示
- Docker Composeプロジェクトの ps / logs / restart
- 操作履歴
- 監査ログ
- allowlist による操作対象制限

### 6.2 対象外とするもの

MVPでは以下を対象外とする。

- ログイン認証
- 複数ユーザー管理
- インターネット公開
- Docker image 削除
- Docker volume 削除
- docker system prune
- 任意コマンド実行
- Webターミナル
- docker exec
- Kubernetes管理
- 複数ホスト管理
- 会社用の権限管理・監査設計

## 7. 基本方針

- WebUIは見やすさを重視する
- 操作よりも状態確認を中心にする
- 危険操作は実装しない
- stop / restart は確認ダイアログを必須にする
- down / delete / prune 系はMVPでは実装しない
- システム停止時もSSHから復旧できる構成にする
- WebUI/APIはrootで動かさない
- ホスト操作は専用Agentに集約する

## 8. 機能要件

### 8.1 ダッシュボード

- ホスト名を表示する
- OS情報を表示する
- 稼働時間を表示する
- CPU使用率を表示する
- メモリ使用率を表示する
- ディスク使用率を表示する
- Docker稼働状況を表示する
- systemd重要サービスの状態を表示する
- 異常状態のサービス・コンテナを強調表示する

### 8.2 systemd管理

- allowlistに登録されたunitのみ表示する
- unit名、表示名、状態、説明を表示する
- statusを取得できる
- startを実行できる
- stopを実行できる
- restartを実行できる
- journalctlログを表示できる
- stop / restart 前に確認ダイアログを表示する

### 8.3 Dockerコンテナ管理

- Dockerコンテナ一覧を表示する
- コンテナ名、イメージ、状態、稼働時間、ポートを表示する
- allowlistに登録されたコンテナのみ操作可能にする
- startを実行できる
- stopを実行できる
- restartを実行できる
- docker logsを表示できる
- stop / restart 前に確認ダイアログを表示する

DockerはDocker Engine APIおよびSDKを提供しており、Docker daemonとやり取りする公式の仕組みがある。実装ではCLI文字列結合よりも、SDKまたは限定されたコマンド実行を優先する。:contentReference[oaicite:1]{index=1}

### 8.4 Docker Compose管理

- 登録済みComposeプロジェクトを一覧表示する
- プロジェクト名、パス、状態、構成サービス数を表示する
- psを表示できる
- logsを表示できる
- restartを実行できる
- MVPでは compose down は実装しない
- MVPでは volume 削除を伴う操作は実装しない

Docker Composeでは depends_on により起動・停止順を制御できる。依存関係のあるサービスではhealthcheckも考慮する。:contentReference[oaicite:2]{index=2}

### 8.5 ログ閲覧

- systemd journalログを表示できる
- Docker logsを表示できる
- Compose logsを表示できる
- 表示行数を選択できる
- 自動更新ON/OFFを切り替えられる
- エラーらしい行を視覚的に強調する

### 8.6 操作履歴

- 実行日時
- 対象種別
- 対象名
- 操作内容
- 結果
- 実行時間
- エラーメッセージ

を表示する。

### 8.7 監査ログ

ログイン認証は実装しないが、以下を記録する。

- 操作日時
- 操作種別
- 操作対象
- 実行元IPアドレス
- User-Agent
- 成功/失敗
- エラー内容

## 9. 非機能要件

### 9.1 可用性

- WebUI停止時もSSHで復旧可能であること
- Agent停止時はWebUIに異常表示すること
- DB停止時でもアプリ全体が無限再起動しないこと

### 9.2 性能

- ダッシュボード初期表示は3秒以内を目標とする
- ログ取得は最大行数を制限する
- 自動更新間隔は最短5秒以上とする

### 9.3 安全性

- 任意コマンド実行を禁止する
- allowlist未登録対象への操作を禁止する
- 削除系操作を禁止する
- WebUI/APIをrootで実行しない
- Docker socketの扱いに注意する
- 外部公開を前提にしない

### 9.4 保守性

- 設定ファイルで管理対象を変更できること
- READMEに起動・停止・復旧方法を記載すること
- docs配下に仕様書を配置すること

## 10. MVP完了条件

- Docker ComposeでWeb/API/DBを起動できる
- WebUIからダッシュボードを表示できる
- systemd unitの状態を表示できる
- Dockerコンテナの状態を表示できる
- Docker logsを表示できる
- allowlist登録済み対象だけrestartできる
- 操作履歴を確認できる
- 監査ログを確認できる
- 削除系操作が実装されていない
- 任意コマンド実行ができない
```
