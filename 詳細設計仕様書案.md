# Ubuntu Ops Control Console 詳細設計仕様書

## 1. 全体構成

本システムは以下の構成とする。

```text
Browser
  ↓
Web UI
  ↓
API Server
  ↓
Host Control Agent
  ├─ systemd control
  ├─ Docker control
  ├─ Docker Compose control
  └─ log collector

PostgreSQL
  ├─ operation_history
  ├─ audit_logs
  └─ managed_targets
````

## 4. コンポーネント設計

### 4.1 WebUI

役割:

* 状態表示
* 操作ボタン表示
* ログ表示
* 操作履歴表示
* 確認ダイアログ表示

WebUIはホスト操作を直接実行しない。

### 4.2 API Server

役割:

* WebUI向けAPI提供
* Agent呼び出し
* 操作履歴保存
* 監査ログ保存
* allowlist確認
* 入力値バリデーション

### 4.3 Host Control Agent

役割:

* systemd状態取得
* systemd操作実行
* Docker状態取得
* Docker操作実行
* Docker Compose状態取得
* ログ取得

systemdはsystem/service managerとしてサービスを管理する仕組みであり、D-Bus API経由でも操作できる。MVPでは実装難易度を考慮し、限定された安全なコマンド実行でもよい。([フリーデスクトップ][2])

### 4.4 PostgreSQL

役割:

* 操作履歴保存
* 監査ログ保存
* 管理対象メタデータ保存
* UI設定保存

## 5. docker-compose.yml構成

```yaml
services:
  web:
    build: ./apps/web
    ports:
      - "127.0.0.1:3000:3000"
    environment:
      - API_BASE_URL=http://api:8000
    depends_on:
      - api

  api:
    build: ./apps/api
    ports:
      - "127.0.0.1:8000:8000"
    environment:
      - DATABASE_URL=postgresql://uocc:uocc@db:5432/uocc
      - AGENT_BASE_URL=http://host.docker.internal:8787
    depends_on:
      db:
        condition: service_healthy

  db:
    image: postgres:16
    environment:
      - POSTGRES_USER=uocc
      - POSTGRES_PASSWORD=uocc
      - POSTGRES_DB=uocc
    volumes:
      - uocc_pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U uocc"]
      interval: 10s
      timeout: 5s
      retries: 5

volumes:
  uocc_pgdata:
```

## 6. Agent起動方式

Agentはsystemd serviceとしてUbuntuホスト側で起動する。

```ini
[Unit]
Description=Ubuntu Ops Control Console Agent
After=network.target docker.service
Requires=docker.service

[Service]
Type=simple
User=uocc-agent
Group=uocc-agent
ExecStart=/opt/uocc-agent/uocc-agent
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

## 10. 操作フロー

```text
1. WebUIで操作ボタン押下
2. 確認ダイアログ表示
3. APIへ操作要求
4. APIでallowlist確認
5. APIで操作履歴をrunningとして登録
6. Agentへ操作依頼
7. Agentが対象種別ごとに操作実行
8. Agentが結果を返却
9. APIがoperation_historyを更新
10. APIがaudit_logsへ記録
11. WebUIに結果を表示
```

## 11. 禁止操作

MVPでは以下の操作を実装しない。

* docker rm
* docker rmi
* docker volume rm
* docker system prune
* docker compose down -v
* docker exec
* 任意shell実行
* systemctl disable
* systemctl mask
* systemctl daemon-reload
* reboot
* shutdown

## 12. エラーハンドリング

| エラー          | 表示                           |
| ------------ | ---------------------------- |
| Agent未応答     | Agent is not reachable       |
| allowlist未登録 | Target is not allowed        |
| 権限不足         | Operation is not permitted   |
| Docker未起動    | Docker daemon is not running |
| systemd操作失敗  | systemd operation failed     |
| ログ取得失敗       | Failed to fetch logs         |

## 13. ログ取得制限

* 初期表示は直近200行
* 選択可能行数は100 / 200 / 500 / 1000
* 1000行を超える取得は禁止
* 自動更新は5秒以上
* ログ全文ダウンロードはMVP対象外
